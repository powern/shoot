# План модернізації 3D-рушія для підтримки Doom II MAP01

## Поточний стан рушія (3Dzavr)

### 1. Завантажувач OBJ
**Файл:** `engine/utils/ResourceManager.cpp:101-166`

**Поточний стан:** Власний спрощений формат з підтримкою лише `v`, `f` (3 індекси), `o`, `g`, `s`, `usemtl` (ігнорується) та кастомного `m ID R G B A`.

**Що НЕ підтримується:**
- `vt` (UV-координати) — відсутні
- `vn` (нормалі) — не читаються, нормалі перераховуються через cross product
- `mtllib` — не підтримується
- `usemtl` — парситься але ігнорується (матеріал береться з `g`)
- `f v1/vt1 v2/vt2 v3/vt3` — не підтримується
- `f v1//vn1 v2//vn2 v3//vn3` — не підтримується
- Чотирикутники та полігони >3 вершин — не підтримуються
- Негативні індекси — не підтримуються
- Рядки >127 символів — мовчки обрізаються (буфер `char line[128]`)
- Коментарі — ігноруються (проходять через всі `if`)
- Помилки — не обробляються (мовчки пропускаються)

### 2. Структури Triangle, Mesh, матеріали

**Triangle** (`engine/Triangle.h`):
```
Vec4D _points[3];    // 3 вершини (однорідні координати)
Vec3D _normal;       // нормаль трикутника (flat shading)
sf::Color _color;    // колір RGBA на трикутник
```
- **Немає UV-координат** — текстурування неможливе
- **Немає per-vertex normal** — тільки flat shading
- Колір зберігається як `sf::Color` (uint8 RGBA)
- Нормаль обчислюється в конструкторі: `(v1-v0) × (v2-v0)`

**Mesh** (`engine/Mesh.h`):
```
std::vector<Triangle> _tris;    // всі трикутники меша
sf::Color _color;               // колір за замовчуванням
bool _visible;                  // видимість
double _boundingRadius;         // радіус для frustum culling
```
- Матеріали — це тільки кольори (`sf::Color`), немає текстур
- Матеріали з OBJ (`m`) зберігаються в `std::map<string, sf::Color>` в ResourceManager

### 3. Завантаження текстур
**Файл:** `engine/utils/ResourceManager.cpp:22-47`

- Використовує `sf::Texture::loadFromFile()`
- Кешування через `std::map<string, shared_ptr<sf::Texture>>`
- Відсутня текстура: `return nullptr`, лог помилки
- Всі текстури: `setRepeated(true)` (тайлинг)
- Формати: bmp, png, tga, jpg, gif, psd, hdr, pic, pnm
- Текстури використовуються ТІЛЬКИ для 2D спрайтів (меню, HUD, зброя)
- 3D-рендеринг не використовує текстури взагалі

### 4. Рендеринг трикутників
**Два шляхи:**

**OpenGL** (`Engine.cpp:72-85`, `Screen.cpp:189-243`):
- `glDrawArrays(GL_TRIANGLES)` з interleaved vertex+color масивом (7 float на вершину: xyz + rgba)
- `GL_TEXTURE_COORD_ARRAY` вимкнено
- `GL_CULL_FACE` ввімкнено, front face = CCW
- `GL_DEPTH_TEST` ввімкнено

**Software** (`Camera.cpp:11-87`, `Screen.cpp:123-135`):
- `Camera::project()`: model transform → backface cull → view transform → frustum clip → lighting → project (SP matrix) → perspective divide
- `Screen::drawTriangle()`: SFML `sf::Triangles` з кольором (CPU rasterizer)

### 5. Z-buffer
- **OpenGL шлях**: апаратний Z-buffer (12-bit, `glClearDepth(1.0)`, `glDepthFunc(GL_LESS)`)
- **Software шлях**: **НЕМАЄ Z-buffer**. Використовується painter's algorithm: `Camera::sorted()` сортує трикутники від далеких до близьких за max Z

### 6. Backface culling
- **OpenGL**: `glCullFace(GL_BACK)` з `glFrontFace(GL_CCW)`
- **Software**: `Camera::project():36-40` — `dot(normal, centroid - cameraPos) > 0` = backface. Працює в world space.

### 7. Clipping біля камери
- 6 clip-площин: near, far, left, right, top, bottom (задаються в `Camera::init()`)
- near plane: `ZNear = 0.01`, far plane: `ZFar = 5000.0`
- `Plane::clip()` — Sutherland-Hodgman: для 1 вершини всередині → 1 трикутник, для 2 → 2 трикутники
- near-plane clipping **коректно обробляє частково видимі трикутники**

### 8. Система камери та руху гравця
**Камера** (`Camera.h/cpp`):
- Успадковує `Object` (має позицію, кут, transform)
- `invModel()` = `View(model())` — view matrix
- `_SP = ScreenSpace * Projection` — комбінована матриця

**Гравець** (`PlayerController.cpp`):
- WASD = translate вздовж left/lookAt з `WALK_SPEED=10`
- Mouse X = yaw (rotate Y), Mouse Y = pitch (rotateLeft, clamped ±90°)
- Space = jump: `addVelocity(0, sqrt(2*35*3), 0)`
- Shift = slow-mo (ділить швидкість на 5, гравітацію на 25)
- Прив'язка до `deltaTime` (frame-rate dependent movement)

### 9. Колізії
**Система** (`engine/physics/`):
- **Broadphase**: spatial grid (`PHYSICS_GRID_SIZE=10.0`)
- **Narrowphase**: GJK (Gilbert-Johnson-Keerthi) + EPA (Expanding Polytope Algorithm)
- **Integration**: Semi-implicit Euler (`v += a*dt`, `x += v*dt`)
- **Response**: імпульсна з restitution + Coulomb friction
- **Static bodies**: `setStatic(true)` для карти (zero inverse mass)
- **HitBox**: simple (8-вершинний AABB) або detailed (всі унікальні вершини меша)
- **Бар'єри**: трикутники карти перевіряються через GJK, але не індивідуально

### 10. Система координат
- **Right-handed** (X×Y = Z, перевірено у Vec3D.cpp:84-88)
- **Y-up**: gravity = `(0, -35, 0)`
- **Z-forward**: `lookAt()` = колонка Z transform matrix = forward direction
- **X-left**: `left()` = колонка X = left direction (при identity: +X = left, -X = right)
- **Одиниці**: 1 unit ≈ 1 метр (висота гравця = 1.8, jump height = 3)
- **View space**: камера дивиться в +Z, near plane при Z=ZNear, far при Z=ZFar
- OpenGL frustum: `glFrustum(-ratio, ratio, -1.f, 1.f, 1.0f, 500.f)`

### 11. Прозорість матеріалів
- `sf::Color` має alpha-канал
- У software path alpha передається в SFML, який підтримує alpha blending
- У OpenGL path: код для `GL_BLEND` закоментовано (Screen.cpp:203-208)
- Прозорі трикутники не сортуються окремо — всі трикутники сортуються разом (painter's algorithm)

### 12. Обмеження на кількість
- **Буфер рядка**: 128 символів — при довгих рядках OBJ (>127) — обрізання
- **Вершини**: `std::vector<Vec4D>` — без обмежень (крім пам'яті)
- **Трикутники**: `std::vector<Triangle>` — без обмежень
- **Об'єкти**: `std::vector<shared_ptr<Mesh>>` — без обмежень
- **Матеріали**: `std::map<string, sf::Color>` — без обмежень
- **OpenGL buffer**: `glFloatArray()` алочує `7 * 3 * tris.size()` float — переалоковується при null (не на кожному кадрі)

## План реалізації

### Етап 1. Повноцінний завантажувач OBJ
Розширити поточний завантажувач для підтримки стандартного Wavefront OBJ.

**Потрібно:**
- Замінити буфер `char line[128]` на `std::string` (немає обмеження довжини)
- Додати парсинг `vt u v` (UV-координати)
- Додати парсинг `vn x y z` (нормалі)
- Додати парсинг `f` у всіх форматах:
  - `f v1 v2 v3`
  - `f v1/vt1 v2/vt2 v3/vt3`
  - `f v1//vn1 v2//vn2 v3//vn3`
  - `f v1/vt1/vn1 v2/vt2/vn2 v3/vt3/vn3`
- Додати підтримку негативних індексів (OBJ convention: -1 = останній)
- Додати `mtllib` (підключення MTL-файлу)
- Додати `usemtl` (вибір матеріалу)
- Створювати фінальні вершини за комбінацією position+uv+normal (vertex deduplication)
- Зберегти сумісність з кастомним `m ID R G B A`
- Додати обробку помилок з номером рядка і назвою файлу
- Додати підтримку коментарів `#` та порожніх рядків
- Додати `s off` / `s 1` (згладжування, ігнорувати але не падати)

**Файли для змін:** `engine/utils/ResourceManager.cpp`, `engine/utils/ResourceManager.h`

### Етап 2. Тріангуляція полігонів
Готовий OBJ може містити не лише трикутники.

**Потрібно:**
- Підтримати грані з трьома і більше вершинами
- Перетворювати їх на трикутники
- Не ламати порядок вершин (winding order)
- Для простих опуклих полігонів — triangle fan
- Для увігнутих — ear clipping алгоритм
- Окремо перевірити підлоги та стелі Doom (можуть бути увігнутими)

**Файли для змін:** `engine/utils/ResourceManager.cpp`

### Етап 3. Підтримка MTL-матеріалів
Додати читання стандартних `.mtl` файлів.

**Мінімально підтримати:**
- `newmtl` — назва матеріалу
- `Kd` — diffuse color (RGB)
- `Ka` — ambient color (якщо використовується)
- `d` / `Tr` — alpha (прозорість)
- `map_Kd` — diffuse texture
- `map_d` — alpha texture (за можливості)

**Структура матеріалу:**
- назва
- базовий колір (sf::Color)
- alpha
- посилання на текстуру (shared_ptr<sf::Texture>)
- ознака прозорості (bool)
- стан завантаження текстури

**Правила:**
- Якщо MTL або текстура відсутні — не падати, використовувати fallback-матеріал (шахова або однотонна текстура), вивести попередження
- Шляхи до MTL і текстур — відносно каталогу OBJ/MTL, а не робочого каталогу
- Обробляти `/` і `\` у шляхах, пробіли в назвах, різний регістр розширень
- Повторне використання текстури кількома матеріалами (кешування)

**Файли для змін:** `engine/Triangle.h`, `engine/Triangle.cpp`, `engine/Mesh.h`, `engine/Mesh.cpp`, новий `engine/Material.h`

### Етап 4. UV-координати та текстурування

**Потрібно:**
- Додати UV-координати до структури вершини / Triangle
- Додати perspective-correct interpolation (1/z)
- Інтерполяція UV з використанням 1/z
- Clipping з правильним перерахунком UV
- Відсікання трикутників біля near plane з UV
- Повторення текстури за межами діапазону 0–1 (repeat/clamp)
- Вертикальне перевертання V (OBJ vs image convention)
- Коректне текстурування після тріангуляції

**Software path:**
- Додати текстурування в `Screen::drawTriangle()` (використовувати `sf::Vertex` з texcoords)
- Або реалізувати власний rasterizer з texture mapping

**OpenGL path:**
- Додати `GL_TEXTURE_COORD_ARRAY` в `prepareToGlDrawMesh()`
- Додати текстуру в `glDrawMesh()`
- Змінити формат вершини з `[x,y,z,r,g,b,a]` на `[x,y,z,u,v,r,g,b,a]` (9 float)

**Важливо:** Не використовувати лінійну інтерполяцію UV у screen space — на довгих стінах Doom буде викривлення.

**Файли для змін:** `engine/Triangle.h/cpp`, `engine/Screen.h/cpp`, `engine/Camera.cpp`, `engine/Mesh.h/cpp`

### Етап 9. Колізії гравця

**Не можна** використовувати всі трикутники карти для наївної перевірки на кожному кадрі.

**Перша реалізація:**
- Додати гравцеві capsule або cylinder collider
- Окремо зберігати: позиція ніг, висота, радіус, висота кроку
- Зіткнення зі стінами + знаходження підлоги
- Не провалюватися крізь підлогу, не проходити крізь стіни
- Підйом на невисокі сходинки
- Ковзання вздовж стіни
- Не застрягати на стиках трикутників

**Прискорення:**
- uniform grid, spatial hash, BVH або інша структура
- Перевіряти лише трикутники поблизу гравця
- Не прив'язувати до частоти кадрів (fixed timestep)

## Мінімальний результат першої ітерації

1. MAP01 завантажується без падіння
2. Геометрія відображається у правильній орієнтації
3. Підлога, стеля, стіни не зникають через неправильний winding
4. Камера знаходиться всередині карти
5. Near-plane clipping працює
6. Z-buffer працює без критичних артефактів
7. Гравець не проходить крізь основні стіни
8. Гравець не провалюється крізь підлогу
9. Старий `map1.obj` продовжує працювати

## Другий результат
1. Завантажується MTL
2. Завантажуються текстури
3. UV відображаються правильно
4. Текстури не спотворюються на довгих стінах
5. Матеріали кешуються
6. Відсутні текстури → fallback

## Третій результат
1. Chunks + frustum culling
2. Оптимізація колізій (spatial grid / BVH)
3. Стабільний FPS
4. Внутрішній оптимізований формат для швидкого завантаження

## Важливі обмеження
- Не переписувати рушій повністю
- Не ламати старі карти
- Не вимикати глобально Z-buffer, clipping або backface culling
